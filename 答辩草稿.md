# 答辩草稿与可能问题

## 一、项目介绍稿

各位老师好，我的项目是一个基于 Windows 本地部署的电子商务系统。系统采用前后端分离架构，前端使用 React 和 Vite，后端使用 Spring Boot，数据存储使用 MySQL，缓存使用 Redis，图片和视频等商品资料使用 MinIO 对象存储。

系统主要实现了用户端和管理端两部分功能。用户端包括首页展示、商品列表、商品详情、购物车、注册登录、结算下单和模拟支付；管理端包括商品管理、分类管理、广告管理、订单管理和用户管理。

在部署方式上，我将项目整理成适合 Windows 本地展示的形式。基础依赖通过 Docker Compose 启动，包括 MySQL、Redis 和 MinIO；后端和前端可以直接在 IntelliJ IDEA 中启动，便于展示代码结构、运行日志和调试过程。

## 二、演示流程稿

我的演示流程分为六步：

1. 首先打开 Docker Desktop。
2. 在 IDEA 中打开项目根目录。
3. 在 IDEA Terminal 执行 `.\scripts\start-local-deps.ps1`，启动 MySQL、Redis 和 MinIO。
4. 运行后端主类 `MallApplication`。
5. 进入 `frontend` 目录执行 `npm run dev` 启动前端。
6. 浏览器访问 `http://localhost:5173`，使用 `admin / admin123` 登录后台。

后端默认连接本地服务：

- MySQL：`localhost:3307`
- Redis：`localhost:6379`
- MinIO：`http://localhost:9000`

MySQL 使用 `3307` 是因为很多 Windows 机器本地已经安装了 MySQL，占用了默认的 `3306`，所以项目主动避开端口冲突。

## 三、系统架构说明

系统采用典型的前后端分离架构：

- 前端负责页面展示、路由跳转、表单交互和调用后端 API。
- 后端负责业务逻辑、权限校验、数据处理和接口返回。
- MySQL 存储用户、商品、订单、购物车等核心业务数据。
- Redis 存储验证码和部分缓存数据，例如商品详情缓存。
- MinIO 用于存储后台上传的商品图片和视频，提供类似 S3 的对象存储能力。

请求链路是：浏览器访问前端，前端通过 `/api` 代理调用 Spring Boot 后端，后端再访问 MySQL、Redis 和 MinIO。

## 四、核心功能说明

### 1. 用户注册和登录

用户注册时可以通过邮箱验证码完成验证。登录成功后，后端会生成 JWT，前端把 JWT 存到本地，后续请求通过请求头携带：

```text
Authorization: Bearer token
```

后端使用过滤器解析 JWT，并将当前用户信息放入上下文。

### 2. 商品浏览

商品列表支持分类筛选和关键字搜索。商品数据来自 MySQL，商品封面来自商品资料表中的图片数据。

### 3. 购物车和下单

用户可以把商品加入购物车，修改数量，勾选商品后结算。下单时后端会生成订单和订单明细，并扣减库存。

### 4. 后台管理

管理员可以管理商品、分类、广告、订单和用户。后台接口通过角色权限控制，只有 `ADMIN` 角色可以访问。

### 5. 文件上传

商品图片和视频上传到 MinIO。后端保存文件 URL，前端通过 URL 展示资源。

## 五、关键技术点

### 1. 为什么使用 JWT？

JWT 适合前后端分离项目。后端登录成功后签发 token，前端后续请求携带 token，后端不需要保存传统 session，接口更容易扩展。

### 2. 为什么使用 Redis？

Redis 用于验证码缓存和商品详情缓存。验证码有过期时间，Redis 可以天然支持 TTL；商品详情缓存可以减少数据库查询压力。

### 3. 为什么使用 MinIO？

商品图片、视频这类文件不适合直接存进数据库。MinIO 是兼容 S3 协议的对象存储，适合本地部署和演示，也方便后续切换到云对象存储。

### 4. 为什么使用 Docker Compose？

本地环境需要 MySQL、Redis、MinIO。手动安装这些依赖成本高，容易出现端口、版本、编码问题。Docker Compose 可以统一启动、停止和重建依赖环境。

### 5. 为什么 MySQL 使用 3307？

Windows 本地经常已经安装 MySQL，占用 `3306`。为了避免冲突，项目把 Docker MySQL 暴露到 `3307`，容器内部仍然是 `3306`。

### 6. 如何解决中文乱码？

项目数据库和表使用 `utf8mb4`。初始化数据导入时必须使用：

```text
mysql --default-character-set=utf8mb4
```

否则中文可能在写入数据库时就变成乱码。项目已经通过 `sql/init-local.sh` 固定了正确导入方式。

### 7. 邮箱验证码如何实现？

后端使用 Spring Mail，通过 SMTP 发送验证码。验证码会存入 Redis，并设置过期时间。用户提交验证码时，后端从 Redis 中取出并校验。

需要配置：

```text
MAIL_HOST
MAIL_PORT
MAIL_USERNAME
MAIL_PASSWORD
```

其中 `MAIL_PASSWORD` 是 SMTP 授权码，不是邮箱登录密码。

## 六、可能被问到的问题

### Q1：你的项目为什么改成本地部署？

答：本项目用于本地展示和答辩，Windows 本地部署更稳定、可复现，也方便老师查看源码和运行效果。MySQL、Redis、MinIO 通过 Docker Compose 启动，避免手动安装依赖；后端和前端通过 IDEA 启动，方便演示代码、日志和调试。

### Q2：前后端是如何交互的？

答：前端使用 Axios 封装请求，统一以 `/api` 为前缀。开发环境下 Vite 会把 `/api` 代理到后端 `http://localhost:8080`。后端 Controller 提供 REST API，并统一返回 `Result` 结构。

### Q3：如何做登录鉴权？

答：用户登录成功后，后端生成 JWT。前端保存 token，并在请求头中携带 `Authorization: Bearer token`。后端通过 `JwtAuthFilter` 解析 token，校验用户身份。后台接口要求用户角色为 `ADMIN`。

### Q4：为什么需要 Redis？

答：Redis 主要用于两类场景：第一是验证码缓存，因为验证码需要短时间有效；第二是商品详情缓存，用来减少重复数据库查询。Redis 的 TTL 能很好地支持验证码过期。

### Q5：为什么不用数据库存图片？

答：图片和视频属于非结构化文件，直接存数据库会影响数据库体积和查询性能。更合理的做法是把文件放到对象存储中，数据库只保存文件 URL。这里使用 MinIO 是因为它可以本地部署，并兼容 S3 协议。

### Q6：下单时如何保证库存正确？

答：下单逻辑在后端服务层完成，会校验购物车商品、SKU、库存和用户身份，然后在事务中创建订单、订单明细并扣减库存。这样可以保证订单数据和库存变更的一致性。

### Q7：你的项目有哪些安全设计？

答：主要包括 JWT 登录态校验、Spring Security 权限控制、后台接口 ADMIN 角色限制、文件上传类型和大小校验、数据库密码和 JWT 密钥通过环境变量配置等。

### Q8：如果邮箱验证码发不出去，怎么排查？

答：先确认 SMTP 服务是否开启，再确认使用的是授权码而不是邮箱登录密码。还要确认 IDEA 运行后端时是否正确传入 `MAIL_HOST`、`MAIL_PORT`、`MAIL_USERNAME`、`MAIL_PASSWORD`。如果 QQ 邮箱返回 `535 Login fail`，通常是授权码错误、服务未开启或账号被限制。

### Q9：为什么之前出现中文乱码？

答：原因是数据库初始化导入时客户端编码不正确，导致中文在写入数据库时已经变成乱码。修复方式是使用 `mysql --default-character-set=utf8mb4` 重新导入数据，并清空旧数据卷。

### Q10：为什么后端曾经报 `Unsupported character encoding 'utf8mb4'`？

答：`utf8mb4` 是 MySQL 字符集名，不是 Java JDBC 的 `characterEncoding` 参数值。JDBC URL 中应该使用 `characterEncoding=UTF-8`，数据库和表本身仍然使用 `utf8mb4`。

### Q11：如何重置整个本地环境？

答：执行：

```powershell
.\scripts\stop-local.ps1 -WithData
.\scripts\start-local-deps.ps1
```

第一条命令会删除 MySQL 和 MinIO 数据卷，第二条命令会重新启动依赖并重新导入初始化数据。

### Q12：项目目前还有哪些不足？

答：目前支付是模拟支付，没有接入真实支付网关；后台没有提供管理员改密功能；邮箱验证码依赖外部 SMTP，演示时可能受邮箱服务限制；系统目前是单机本地部署，没有做分布式高可用。

### Q13：如果后续继续完善，你会怎么做？

答：后续可以接入真实支付网关，增加管理员改密和找回密码功能，增加操作日志和审计功能，完善单元测试和接口测试，并进一步做 Docker 镜像发布和生产环境部署。

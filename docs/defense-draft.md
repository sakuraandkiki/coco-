# 答辩草稿：XX 电子商务系统

## 1. 开场介绍

老师好，我的项目是一个 XX 电子商务系统，主要面向普通用户购物和管理员后台管理两个角色。普通用户可以浏览商品、注册登录、加入购物车、结算下单、查看订单；管理员可以维护商品、分类、广告、订单和用户信息。

本项目采用 Windows 本地部署方式，后端使用 Maven WebApp、Jakarta Servlet 和 JDBC，前端使用 HTML、CSS、jQuery，数据库使用 MySQL，缓存使用 Redis，对象存储使用 MinIO，最终部署到 Tomcat 10。

## 2. 项目架构

整体架构可以分为五层：

1. 前端展示层：`index.html`、`app.css`、`app.js`，使用 jQuery 发送 AJAX 请求。
2. 控制层：`MallServlet` 统一接收 `/api/*` 请求，根据路径分发到不同业务方法。
3. 工具层：`Db` 负责 JDBC 操作，`RedisClient` 负责 Redis，`MailClient` 负责邮箱验证码，`Web` 负责 JSON 响应。
4. 数据层：MySQL 保存用户、商品、分类、广告、购物车、订单等核心数据。
5. 依赖服务层：Redis 保存验证码和缓存，MinIO 保存商品图片、视频等媒体文件。

部署时，MySQL 原生安装在 Windows，Redis 使用 Docker Desktop 启动容器，MinIO 使用 Windows exe 启动，WebApp 部署到 Tomcat。

## 3. 技术选型说明

- 选择 Maven WebApp + Servlet，是为了符合传统 Java Web 项目的课程要求，并且便于部署到 Tomcat。
- 选择 JDBC，是为了直接展示 SQL 和数据库访问过程，便于说明数据库表结构和业务逻辑。
- 选择 jQuery，是为了让前端结构简单，所有页面逻辑集中在一个单页应用中，方便演示。
- 选择 Redis，是因为邮箱验证码天然需要过期时间和发送冷却，商品列表也适合短时间缓存。
- 选择 MinIO，是为了模拟真实项目中的对象存储，支持商品图片和视频扩展。

## 4. 功能模块介绍

### 4.1 普通用户功能

普通用户端包括：首页、商品列表、商品详情、购物车、结算、订单和登录注册。

用户可以在首页查看广告和推荐商品，点击广告可以进入对应商品分类。进入商品详情后可以查看商品图片、视频、商品资料、规格参数和 SKU。用户登录后可以加入购物车，修改购物车数量，结算生成订单，也可以查看订单详情、模拟支付或取消订单。

普通用户注册时需要填写邮箱，系统会发送真实验证码。验证码存入 Redis，设置 5 分钟过期，同时设置 60 秒发送冷却，防止频繁发送。

### 4.2 管理员功能

管理员使用固定账号 `admin / admin123` 登录。管理员登录后看到的是后台管理菜单，而不是普通用户购物菜单。

后台包括：

- 商品分类管理：新增、编辑、禁用，支持筛选。
- 商品管理：新增、编辑、下架，维护商品图片、视频、子信息和 SKU。
- 广告管理：新增、编辑、禁用广告，广告可以关联分类。
- 订单管理：查看订单并修改订单状态。
- 用户管理：查看用户，按角色、用户名、ID 筛选，启用或禁用用户。

## 5. 数据库设计

数据库沿用 `sql/init.sql`，核心表包括：

- `t_user`：保存用户信息，包含用户名、密码、邮箱、角色、状态等字段。
- `t_product_category`：保存商品分类，支持父级 ID 和状态。
- `t_product`：商品主表，保存名称、价格、库存、状态等核心信息。
- `t_product_info1`、`t_product_info2`：保存商品扩展资料。
- `t_product_sku`：保存 SKU 信息。
- `t_product_media`：保存商品媒体地址，支持图片和视频。
- `t_cart_item`：保存购物车数据。
- `t_order`、`t_order_item`：保存订单主表和订单明细。
- `t_ad_category`、`t_advertisement`：保存广告分类和广告内容。

数据库编码使用 `utf8mb4`，导入数据时也使用 `--default-character-set=utf8mb4`，避免中文乱码。

## 6. 关键流程说明

### 6.1 登录流程

用户在前端输入用户名和密码，前端通过 AJAX 调用 `/api/users/login`。后端先判断是否为管理员固定账号，如果是则写入管理员 Session。如果不是管理员，则查询 `t_user` 中角色为普通用户的数据，并用 BCrypt 校验密码。校验成功后保存用户 Session，失败则返回“用户不存在或密码错误”。

### 6.2 注册验证码流程

用户点击发送验证码后，前端调用 `/api/users/send-code`。后端生成 6 位验证码，通过 SMTP 发送到用户邮箱，然后写入 Redis：

- `verify:register:{email}` 保存验证码，5 分钟过期。
- `verify:cooldown:{email}` 保存冷却标记，60 秒过期。

注册提交时，后端读取 Redis 中的验证码并比较，正确后创建用户。

### 6.3 下单流程

用户在购物车选择商品后提交结算，后端读取购物车明细，计算总价，写入 `t_order` 和 `t_order_item`。订单创建成功后清空对应购物车数据。

### 6.4 权限控制

后端对 `/api/admin/*` 接口统一校验 Session 中的 `role` 是否为 `ADMIN`。如果不是管理员，直接返回无权限，避免普通用户访问后台接口。

## 7. 老师可能问的问题与回答

### Q1：为什么使用 Maven WebApp 而不是 Spring Boot？

答：这次项目目标是符合课程中的传统 Java Web 展示要求，所以改成了 Maven WebApp + Servlet。这样可以更直观地展示 Web 容器、Servlet、JDBC、Session、Tomcat 部署等基础知识。

### Q2：普通用户和管理员如何区分？

答：登录成功后，后端会在 Session 中保存用户角色。管理员使用固定账号 `admin / admin123` 登录，角色为 `ADMIN`；普通用户从数据库 `t_user` 中校验，角色为 `USER`。前端根据 Session 返回的角色展示不同菜单，后端也会保护 `/api/admin/*` 接口。

### Q3：为什么注册要使用 Redis？

答：验证码需要自动过期，并且需要限制重复发送。Redis 支持 key 的过期时间，非常适合保存验证码和冷却标记。如果用 MySQL，也能实现，但需要额外清理过期数据，复杂度更高。

### Q4：数据库如何防止中文乱码？

答：数据库创建时使用 `utf8mb4`，SQL 导入时带 `--default-character-set=utf8mb4`，JDBC URL 中指定 `characterEncoding=UTF-8`，页面和响应也设置 UTF-8。

### Q5：商品图片和视频放在哪里？

答：商品媒体信息保存在 `t_product_media` 表，真实文件可以放在 MinIO 中。MinIO 提供类似对象存储的访问方式，前端可以通过 URL 加载图片或视频。

### Q6：项目有哪些安全处理？

答：密码使用 BCrypt 保存和校验，后台接口通过 Session 角色校验，验证码有过期时间和发送冷却。演示环境中管理员是固定账号，正式环境可以改为数据库管理员账号并增加更严格的权限控制。

### Q7：如果 Redis 或 MinIO 没启动会怎样？

答：Redis 没启动会影响验证码和缓存功能；MinIO 没启动会影响新上传媒体文件，但数据库中的已有图片 URL 仍可能显示。部署前需要按 README 检查三个依赖服务。

### Q8：项目不足和后续改进？

答：当前项目重点是课程展示和本地部署。后续可以增加订单支付接口、库存并发扣减、后台权限细分、MinIO 直传、前后端分离版本、统一异常处理和更完整的自动化测试。

4. 普通用户注册，展示邮箱验证码流程。
5. 普通用户登录，加入购物车并结算下单。
6. 查看订单，模拟支付或取消订单。
7. 退出后使用 `admin / admin123` 登录后台。
8. 展示分类、商品、广告、订单、用户管理和筛选功能。
9. 简要说明 MySQL、Redis、MinIO、Tomcat 的本地部署架构。
